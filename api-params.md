## Summary — the useful levers (confirmed)

The feed query is **ProfileCometTimelineFeedRefetchQuery** (doc_id 27631023009915089).
Of its ~45 variables, most are relay feature-flags (`__relay_internal__pv__*`). The
ones that actually control what you get back:

| variable | meaning | confirmed behavior |
|---|---|---|
| `cursor` | pagination pointer | opaque **encrypted** `organic_cursor`; only source of the next page (serial) |
| `count` | page size requested | **ignored** by FB — always returns 3 for this profile feed |
| `beforeTime` | upper time bound (unix s) | **year filter sets this** (2022 → 1672502399 = 2022-12-31T15:59:59.000Z) |
| `afterTime` | lower time bound (unix s) | accepted but UI leaves null; set it yourself to bound a window |
| `feedLocation` | "TIMELINE" | which feed surface |
| `postedBy` | {group:"OWNER"} | owner posts vs tagged/others |
| `omitPinnedPost`, `privacy`, `taggedInOnly` | minor filters | dialog defaults |
| `id` | the profile/page id (100064545286089) | target entity |

**Parallelization insight:** `afterTime`+`beforeTime` together bound a date window.
Set non-overlapping windows (e.g. per year or per month) to run independent,
non-overlapping chains concurrently — the only way around the serial encrypted cursor.
Note the year UI only sets `beforeTime` (so its chains would overlap); setting BOTH
bounds ourselves gives clean partitions.
